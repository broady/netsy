# Netsy V2 Implementation Plan

## What Exists Today

- **Data file format**: `.netsy` file reader/writer with CRC validation (`internal/datafile/`)
- **Protobuf schemas**: Record, FileHeader, FileFooter, Upload (`proto/`)
- **SQLite local database**: schema, insert, find, transaction logic (`internal/localdb/`)
- **S3 client**: chunk/snapshot upload, download, listing, cleanup (`internal/s3client/`)
- **Client API (etcd-compatible)**: Range, Txn, Watch, LeaseGrant, MemberList, Status (`internal/clientapi/`)
- **Peer API**: leader transactions, snapshots (`internal/peerapi/`) — note: misleading name, this is not a gRPC server but the internal domain layer for Primary write logic (LeaderTxn) and snapshot creation, called by `clientapi`
- **Snapshot worker**: (`internal/snapshot/`)
- **Backfill**: (`internal/backfill.go`)
- **Configuration**: init, validation, file loading (`internal/config/`)

## Implementation Phases & Tasks

### Phase 1: Refactor Config & Packages

Single-node Primary already works (mTLS, SQLite, S3 sync writes, Client API, signal handling). This phase cleans up config/naming and restructures internals.

- [ ] Rename/restructure `internal/peerapi/` — it's the Primary domain layer, not a gRPC peer server
- [ ] Implement JSONC cluster config file loading (`--config` flag / `NETSY_CONFIG` env var, required)
- [ ] Implement `--validate <path>` flag to validate a config file and exit
- [ ] Config & naming cleanup:
  - [ ] Rename `InstanceID` -> `NodeID` throughout (config, s3client, peerapi, snapshot) and add validation (lowercase alphanumeric and hyphens only, no leading, trailing, or consecutive hyphens, max 32 characters. for this it could be a general "identifier" validation function, which can be later used for validating the cluster_id)
  - [ ] Remove `InstanceHostname` (defined but unused outside config)
  - [ ] Replace `ReplicationMode` (synchronous/asynchronous string) with quorum config (`-1`, `0`, positive int)
  - [ ] Rename `ListenClientsAddr` -> Client API bind address, add Client API advertise address
  - [ ] Rename `ListenPeersAddr` -> Peer API bind address, add Peer API advertise address
  - [ ] Add s3lect health server bind address + advertise address (HTTPS endpoint used exclusively by s3lect for peer-to-peer leader election health-checking; advertise address registered with s3lect and included in TLS cert)
  - [ ] Add HTTP health endpoint bind address (for `/health`, HTTP only — no advertise address needed)
- [ ] Implement full config struct matching `docs/deployment/config.md` — parse and validate all fields upfront:
  - [ ] `cluster_id` with validation (re-using the identifier validation used for node_id: lowercase alphanumeric and hyphens only, no leading, trailing, or consecutive hyphens, max 32 characters)
  - [ ] `storage.*` (provider, bucket_name, key_prefix, class, encryption, kms_key_id)
  - [ ] `elector.*` (heartbeat_interval, degradation_timeout, deregistration_timeout, primary_prior_timeout)
  - [ ] `replication.*` (quorum, heartbeat_interval, degradation_timeout, chunk_buffer.threshold_size_mb, chunk_buffer.threshold_age_minutes)
  - [ ] `snapshot.*` (threshold_records, threshold_size_mb, threshold_age_minutes)
  - [ ] `compaction_interval`

### Phase 2: Storage Abstraction

- [ ] Abstract object storage behind an interface (currently hardcoded to S3 in `internal/s3client/`), add GCS support

### Phase 3: mTLS Identity Enforcement

- [ ] Implement mTLS authentication/authorization code path for all gRPC servers using certificates with an embedded role (`peer` or `client`):
  - `peer` certificates must verify CN matches `node_id` and O matches `cluster_id`;
  - `client` certificates authenticate external Client API callers without requiring Node identity matching, and 0 matches `cluster_id`

### Phase 4: SQLite Changes

- [ ] Add `PRAGMA synchronous=FULL` (currently missing — only WAL is set. fsync is critical for Receipt durability)
- [ ] Set `PRAGMA user_version` to track schema version; refuse to start if the database version is different than expected (guards against older release using a newer database); if lower, a future release may migrate in-place or wipe and rebuild from object storage

### Phase 5: Node State & Health Endpoint

Define the state enums and health endpoint.

- [ ] Health State enum (Loading -> Healthy -> Degraded) with transitions
- [ ] Elector State enum (Leader -> Follower) with transitions
- [ ] Primary State enum (Replica -> Starting -> Active -> Draining) with transitions
- [ ] `/health` HTTP endpoint driven by Health State (currently uses gRPC health only)

### Phase 6: s3lect Integration

- [ ] s3lect integration for Elector leader election ("first tier" of leader election)

### Phase 7: Peer API Protobuf Definitions

Define the gRPC service and message types used by subsequent phases. Proto files only — implementation is done in each phase that uses them.

- [ ] Define `Elector` service proto (RegisterNode, DeregisterNode, GetClusterState, SendHeartbeat)
- [ ] Define `Node` service proto (PushClusterState, GetNodeState, GetGlobalMinWatchRevision, SendCompactionNotice)
- [ ] Define `Primary` service proto (Follow, SendHeartbeat)
- [ ] Define shared message types (NodeState, ClusterState, HealthState, ElectorState, PrimaryState, etc.)

### Phase 8: Service Discovery & Node Registration

- [ ] Define node registration file format (JSON under `nodes/{node_id}.json` in object storage, containing Peer and Client advertise addresses)
- [ ] Define durable member ID file format (`members.json` in object storage, containing `cluster_id` and `member_id -> node_id` mappings)
- [ ] Service Discovery — Node registration in object storage (`nodes/` prefix)
- [ ] Elector-managed member ID allocation/reuse in `members.json` using object storage `If-Match`
- [ ] Implement RegisterNode / DeregisterNode RPCs (Node ↔ Elector)
- [ ] Elector in-memory Node map (populated from object storage, storing each Node's advertise addresses and stable etcd `member_id`, updated by direct registrations)
- [ ] Elector node-map bootstrap: on becoming Elector, immediately load `members.json` and all node files from object storage; accept direct registrations/deregistrations concurrently during this load (do not overwrite a direct deregistration with a stale object storage file); do not perform Primary election until initial load completes
- [ ] Node deregistration (graceful + auto-deregistration via `elector.deregistration_timeout`)

### Phase 9: Graceful Shutdown

Now that deregistration RPCs exist (Phase 8), implement the full shutdown sequence. Chunk Buffer flush (step 2) is a no-op until Phase 16 adds the Chunk Buffer (before Phase 17: Quorum Transactions).

- [ ] Graceful shutdown sequence:
  1. (Primary only) Move Primary State to `Draining` — stop accepting writes
  2. (Primary only) Flush Chunk Buffer to object storage
  3. Move Health State to `Degraded`
  4. Deregister with Elector + delete registration file from object storage
  5. Close gRPC connections
  6. Exit
- [ ] Primary self-degradation: if the Primary itself becomes Degraded (e.g. cannot reach Elector), transition to `Draining` -> flush -> give up leadership -> restart as `Replica` with `Loading` Health State

### Phase 10: Heartbeats

- [ ] Heartbeat RPC — Elector receives NodeState from each Node
- [ ] Heartbeat routing: always send to Elector on cadence; send to Primary only when no Receipt sent within cadence; single path when Elector == Primary
- [ ] Heartbeat-based degradation (2 missed heartbeats -> Degraded by Elector/Primary)

### Phase 11: Primary Election & Cluster State

- [ ] Primary leader election ("second tier") — the full election process (steps 1-7)
- [ ] Cluster State push — triggered immediately on Elector/Primary in-memory state change; push to Primary first, then all other Nodes
- [ ] Node-side split-brain guard: detect two concurrent Elector connections

### Phase 12: Replication Stream & `committed_revision` Tracking

- [ ] Replication gRPC bidirectional stream (`Follow` RPC)
- [ ] Primary sends Records and Commits (`committed_revision`) to Replicas
- [ ] Replicas track `committed_revision` from the replication stream
- [ ] Replicas send Receipts (with embedded Heartbeats) to Primary
- [ ] Standalone Heartbeat fallback (when no Receipt sent within cadence)
- [ ] Replication stream state validation: Replicas must reject data from a node whose Primary State is `Replica`; Primary must accept connections while `Starting`, `Active`, or `Draining`
- [ ] Replica: serve Range (read) requests up to `committed_revision`
- [ ] Watch delivery gated by `committed_revision`

### Phase 13: Node Loading & Backfill

Note: this process is designed such that a single node setup skips Primary discovery/streaming steps, backfills from object storage only, then becomes Elector -> Primary, as noted with "if one exists" below.

- [ ] Determine current Elector via s3lect (if one exists)
- [ ] Register in object storage (`nodes/` prefix) if not already registered (using If-Match/ETag)
- [ ] Register directly with the current Elector (if one exists), which allocates or reuses the Node's stable etcd `member_id` in `members.json`
- [ ] Initialise SQLite database (create schema if new, set/check `PRAGMA user_version`, refuse to start on schema-version mismatch)
- [ ] If `records` table is empty, download and import latest Snapshot from object storage (if one exists)
- [ ] Connect to Elector to discover current Primary (if one exists)
- [ ] Connect to Primary (if one exists) and establish replication stream (begin receiving Records)
- [ ] Backfill gap: fetch Chunk files from object storage to fill records between latest Snapshot and newly replicated Records
- [ ] Integrity check: verify no missing data; if gaps found, truncate table and restart backfill
- [ ] Transition Health State from `Loading` to `Healthy`

### Phase 14: Replica Health Tracking & Degradation

- [ ] Tracking Replicas for quorum (health map, Receipt count, atomic fields per docs)
- [ ] Replica self-degradation (failed Receipt/Heartbeat after 1 retry)
- [ ] Replica self-degradation on `committed_revision` lag: on receiving a `committed_revision` higher than own latest revision, defer a check 2 seconds later; if still behind at that point, transition to `Degraded` ("2 seconds" should be specified in a constant for now)

### Phase 15: Primary Startup & Failover

Newly elected Primary must complete preflight checks before accepting any writes.

- [ ] Primary preflight checks (Starting state):
  1. Download latest chunks from object storage
  2. If the `compactions` table is empty, seed it by inserting the Compaction Revision implied by existing `records` rows with `compacted_at` already set
  3. Set `committed_revision` to latest revision
  4. Push `committed_revision` to Replicas and accept replication stream connections while `Starting`
  5. Transition to Active

### Phase 16: Chunk Buffer

- [ ] Chunk Buffer implementation: in-memory buffer for async object storage writes with size threshold + time interval flush, immediate flush on `Draining`, transition to `Draining` if buffer full and cannot flush

### Phase 17: Quorum Transactions

- [ ] Quorum Transaction logic (Path 2) — send Record, wait for Receipts, advance `committed_revision`
- [ ] Path switching (Path 1 <-> Path 2) based on healthy Replica count
- [ ] Primary fallback to sync object storage on degradation
- [ ] Object storage transaction failure path: retry once -> rollback SQLite -> transition to `Draining` -> continue retrying with exponential backoff -> on success give up leadership and restart as Replica
- [ ] Rollback on timeout / insufficient Receipts
- [ ] On quorum rollback: immediately trigger async flush of previously buffered records to object storage
- [ ] Same-revision retry: after quorum rollback, client retry via Path 1 reuses the same revision number; Replicas overwrite tentative records above `committed_revision`

### Phase 18: Replica Write Proxying

- [ ] Replica: proxy Txn (write) requests to Primary

### Phase 19: Client API Cluster Topology & Status

- [ ] `MemberList`: use stable etcd `member_id`s from Elector-managed `members.json` (`member_id -> node_id`)
- [ ] `MemberList`: proxy to the Elector when the local Node is not the Elector; Elector answers from its in-memory node map
- [ ] `MemberList`: return registered Nodes with correct Client/Peer advertise addresses from service discovery / Cluster State
- [ ] `Status`: answer locally (do not proxy) for the responding Node using local SQLite metrics/state and current Cluster State
  - [ ] Set `leader` to the current Primary's stable etcd `member_id` (or `0` if no Primary is known)
  - [ ] Populate `db_size` / `db_size_in_use` from the responding Node's local SQLite database and keep `version` as the etcd compatibility version string
  - [ ] Populate `errors` from the responding Node's local Health State (`Loading` / `Degraded`), otherwise return an empty list
  - [ ] Keep `raft_index`, `raft_term`, and `raft_applied_index` as `0`, and keep `is_learner=false`, documenting that these are intentionally unsupported/static because Netsy does not use Raft and has no learner role

### Phase 20: Watches & Compaction

- [ ] Watch minimum revision tracking (per-Node, per-Watch)
- [ ] Peer API endpoint for global min Watch revision
- [ ] SQLite: `compactions` table for storing compaction revisions
- [ ] Compaction scheduling by Primary (periodic, using `compaction_interval` from config)

### Phase 21: Compaction Protocol

- [ ] Compaction notice/confirmation protocol:
  1. Primary retrieves global min watch revision from each Node via Peer API
  2. Primary computes the global minimum across all Nodes as the Compaction Revision
  3. Primary sends compaction notice to every Node with the new Compaction Revision
  4. Each Node confirms no existing Watch has a lower revision, then blocks new Watch requests below this revision
  5. Failed confirmations retried once; if still failing, abort compaction until next interval
- [ ] Insert the Compaction Revision into `compactions` table and replicate the record to all Replicas
- [ ] Compaction execution: async task sets `value = NULL` and `compacted_at` timestamp for records below Compaction Revision (rows are not deleted)
- [ ] Watch admission gating: after accepting a compaction notice, Nodes must reject new Watch requests for revisions below the Compaction Revision
- [ ] Compaction + Snapshot mutual exclusion
- [ ] Update Compact implementation status in docs

## Technical Reference

### Transaction Flow Summary

```
Record (Primary -> Replica) -> Receipt (Replica -> Primary) -> Commit (Primary -> Replica)
```

### Peer API Protobuf Definition

The Primary hosts the Peer gRPC server that Replicas connect to for replication, and the Elector hosts a Peer gRPC server for registration, heartbeats, and cluster state.

### Shared Protobuf Definition

```protobuf
message NodeState {
  string node_id = 1;
  HealthState health_state = 2;
  ElectorState elector_state = 3;
  PrimaryState primary_state = 4;
  int64 latest_revision = 5;
  int64 start_time = 6;  // used during election tiebreaker
}

enum HealthState {
  HEALTH_UNKNOWN = 0;
  HEALTH_LOADING = 1;
  HEALTH_HEALTHY = 2;
  HEALTH_DEGRADED = 3;
}

enum ElectorState {
  ELECTOR_UNKNOWN = 0;
  ELECTOR_FOLLOWER = 1;
  ELECTOR_LEADER = 2;
}

enum PrimaryState {
  PRIMARY_UNKNOWN = 0;
  PRIMARY_REPLICA = 1;
  PRIMARY_STARTING = 2;
  PRIMARY_ACTIVE = 3;
  PRIMARY_DRAINING = 4;
}
```

#### Node Protobuf Definition

```protobuf
// Hosted by every Node (called by Elector/Primary)
service Node {
  rpc PushClusterState(ClusterState) returns (google.protobuf.Empty);
  rpc GetNodeState(google.protobuf.Empty) returns (NodeState);
  rpc GetGlobalMinWatchRevision(google.protobuf.Empty) returns (MinWatchRevisionResponse);
  rpc SendCompactionNotice(CompactionNotice) returns (CompactionConfirmation);
}

message MinWatchRevisionResponse {
  int64 min_revision = 1;
}

message CompactionNotice {
  int64 compaction_revision = 1;
}

message CompactionConfirmation {
  bool confirmed = 1;
}
```

#### Elector Protobuf Definition

```protobuf
// Hosted by the Elector
service Elector {
  rpc RegisterNode(RegisterNodeRequest) returns (RegisterNodeResponse);
  rpc DeregisterNode(DeregisterNodeRequest) returns (google.protobuf.Empty);
  rpc GetClusterState(google.protobuf.Empty) returns (ClusterState);
  rpc SendHeartbeat(NodeState) returns (google.protobuf.Empty);
}

message RegisterNodeRequest {
  string node_id = 1;
  string client_advertise_address = 2;
  string peer_advertise_address = 3;
}

message RegisterNodeResponse {
  ClusterState cluster_state = 1;
}

message DeregisterNodeRequest {
  string node_id = 1;
}

message ClusterState {
  string elector_node_id = 1;
  string primary_node_id = 2;  // empty if no Primary elected
}
```

#### Primary Protobuf Definition

```protobuf
// Hosted by the Primary
service Primary {
  rpc Follow(stream ReplicaMessage) returns (stream PrimaryMessage);
  rpc SendHeartbeat(NodeState) returns (google.protobuf.Empty);
}

// Primary -> Replica (Record or Commit)
message PrimaryMessage {
  oneof payload {
    Record record = 1;
    int64 commit = 2;  // committed_revision
  }
}

// Replica -> Primary (Receipt)
message ReplicaMessage {
  int64 revision = 1;      // the revision being receipted
  NodeState heartbeat = 2;  // embedded heartbeat
}

```

### Node Registration File Format

Written to object storage under `nodes/{node_id}.json` using JSON format.

Example:

```json
{
  "node_id": "node-abc-123",
  "peer_addr": "172.16.0.1:2380",
  "client_addr": "172.16.0.1:2379"
}
```

Fields:

- __node_id__: The unique identifier for this node, matching the filename (with .json extension)
- __peer_addr__: Advertise address Peers use to connect to this node's Peer gRPC server (HTTPS/mTLS)
- __client_addr__: Advertise address Clients use to connect to this node's Client gRPC server (HTTPS/mTLS)

### Members File Format

Written to object storage as `members.json` using JSON format and updated only by the Elector using object storage `If-Match` semantics.

Example:

```json
{
  "cluster_id": "my-cluster",
  "members": {
    "1": "node-abc-123",
    "2": "node-def-456"
  }
}
```

Fields:

- __cluster_id__: The Cluster identifier this member mapping belongs to
- __members__: Map of stable etcd `member_id` values to `node_id`

### SQLite Schema

PRAGMAs (set on every connection):

```sql
PRAGMA journal_mode=WAL;
PRAGMA synchronous=FULL;
```

Tables:

```sql
CREATE TABLE IF NOT EXISTS records (
    revision integer PRIMARY KEY NOT NULL,
    key blob NOT NULL,
    created integer NOT NULL,
    deleted integer NOT NULL,
    create_revision integer NOT NULL,
    prev_revision integer NOT NULL,
    version integer NOT NULL,
    lease integer NOT NULL,
    dek integer NOT NULL,
    value blob,
    created_at text NOT NULL,
    compacted_at text,
    leader_id text NOT NULL,
    replicated_at text
);

CREATE UNIQUE INDEX IF NOT EXISTS records_key_create_rev_prev_rev_uindex
    ON records (key, create_revision, prev_revision);
CREATE INDEX IF NOT EXISTS records_index_key ON records (key);

-- New in V2
CREATE TABLE IF NOT EXISTS compactions (
    compaction_revision integer PRIMARY KEY NOT NULL,
    created_at text NOT NULL
);
```

### Object Storage Key Layout

```
{prefix}/chunks/{revision}.netsy          -- Chunk files (one or more sequential records)
{prefix}/snapshots/{revision}.netsy       -- Snapshot files (complete set of KV records after compaction)
{prefix}/members.json                     -- Stable etcd member ID mappings (`member_id` -> `node_id`)
{prefix}/nodes/{node_id}.json             -- Node registration files
```

### V2 Config Parameters

Configuration is split into three tiers:

- **Per-node settings** are unique to each Node (identity, addresses, TLS cert paths). Set via **environment variables** only.
- **Per-cluster settings** are identical across all Nodes (cluster behaviour, storage, thresholds). Set via a shared config file, pointed at by `NETSY_CONFIG` env var or `--config` flag.
- **Object storage connectivity** may additionally be configured via SDK-specific environment variables. These follow the standard conventions of the cloud provider SDK.

A config file is required. Use `--validate <path>` to validate a config file without starting the server.

#### Per-Cluster Settings (config file only)

Identical across all Nodes — cluster identity, storage, behaviour, and thresholds. Set in the JSONC config file only.

```jsonc
{
  // Cluster identity — must match the Organization (O) field in TLS certificates
  "cluster_id": "my-cluster",

  // Object storage configuration
  "storage": {
    "provider": "s3",       // "s3" or "gcs"
    "bucket_name": "my-netsy-bucket",
    "key_prefix": "",       // optional prefix for all object storage keys
    "class": "STANDARD",    // S3 storage class (STANDARD, STANDARD_IA, etc.)
    "encryption": "AES256"  // server-side encryption (AES256 or aws:kms)
    // "kms_key_id": ""     // only needed when using aws:kms encryption
  },

  // Elector leader election and node lifecycle
  "elector": {
    "heartbeat_interval": "1s",       // how often each Node sends a heartbeat to the Elector
    "degradation_timeout": "3s",      // 2 missed heartbeats within this window → Node marked Degraded
    "deregistration_timeout": "3m",   // auto-deregister Degraded nodes after this duration ("0" to disable)
    "primary_prior_timeout": "5s"     // how long to wait for the previous Primary during election before proceeding
  },

  // Replication stream between Primary and Replicas
  "replication": {
    "quorum": -1,                     // -1 = majority, 0 = disabled (synchronous writes to object storage), positive int = static
    "heartbeat_interval": "500ms",    // send a standalone Heartbeat to Primary when no Receipt sent within this window
    "degradation_timeout": "2s",      // 2 missed Heartbeats/Receipts within this window → Replica excluded from quorum
    // Buffer for async object storage writes
    "chunk_buffer": {
      "threshold_size_mb": 4,         // flush chunk buffer to object storage when it exceeds this size
      "threshold_age_minutes": 1      // flush chunk buffer to object storage after this duration
    }
  },

  // Snapshot creation thresholds
  "snapshot": {
    "threshold_records": 10000,       // create snapshot after N records since last snapshot
    "threshold_size_mb": 10000,       // create snapshot when chunks exceed N MB
    "threshold_age_minutes": 0        // create snapshot after N minutes since last snapshot (0 = disabled)
  },

  // How often the Primary schedules compaction across all Nodes
  "compaction_interval": "5m"
}
```
